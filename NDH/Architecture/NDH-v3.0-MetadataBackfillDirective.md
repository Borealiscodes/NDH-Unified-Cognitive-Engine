## **2. Purpose**

The Metadata Backfill Directive (MBD) establishes the requirement that **all major NDH v3.0 layers** must include a **Metadata Block** now that NDH has:

- a Stability Protocol (NSP)  
- a Data‑Neutrality Layer (DNL)  
- a Creative Engine Runtime Bridge (CERB)  
- a Protocol Enhancement Proposal system (PEP)  

These components introduce **protocol identity**, **governance flags**, **runtime invariants**, and **neutrality constraints**, making metadata blocks **mandatory** for architectural coherence.

---

## **3. Rationale**

### **3.1 Before NSP**  
NDH layers were conceptual and diagrammatic.  
Metadata blocks were optional.

### **3.2 After NSP, DNL, CERB, PEP**  
NDH became a **living architecture** with:

- metabolism  
- immunity  
- circulation  
- evolution  
- governance  
- runtime identity  

Metadata blocks are now required for:

- versioning  
- invariants  
- accessibility declarations  
- governance levels  
- neutrality flags  
- runtime integration markers  
- poverty‑safe compliance  

Without metadata blocks, NDH cannot:

- validate layers  
- enforce ethics  
- maintain accessibility  
- route runtime behavior  
- evolve safely  
- maintain neutrality  
- prevent reality drift

## **5. Metadata Block Specification**

Each layer must include a metadata block with the following minimum fields:

```
<METADATA>
  <version>3.0</version>
  <layer>[LayerName]</layer>

  <governance-level>[1–3]</governance-level>
  <accessibility-invariants>true</accessibility-invariants>
  <data-neutrality>true</data-neutrality>
  <poverty-safe>true</poverty-safe>

  <runtime-integration>true</runtime-integration>
  <jdtr-logging>true</jdtr-logging>

  <invariants>[List of layer-specific invariants]</invariants>
</METADATA>
```

Optional fields may include:

- `<kspace-rank>`  
- `<tensor-operator>`  
- `<geometry-operator>`  
- `<darkmode-operator>`  
- `<creative-mapping>`  
- `<orchestration-hooks>`
