# **NDH Runtime Event Bus**  
*Unified publish/subscribe communication fabric for NDH’s semantic–geometric runtime*





---

## **1. Purpose**

The NDH Runtime Event Bus provides a **centralized, asynchronous, event‑driven communication layer** that allows all NDH subsystems to:

- publish events  
- subscribe to event streams  
- react to state changes  
- coordinate behavior  
- propagate warnings and alerts  
- trigger cross‑subsystem actions  

It is the **runtime nervous system**, enabling NDH to behave like a coherent manifold rather than isolated modules.

---

## **2. Responsibilities**

The Event Bus:

- receives events from all subsystems  
- categorizes events by type and priority  
- routes events to subscribers  
- enforces safety and ordering rules  
- logs event traffic for replay and diagnostics  
- integrates with the **Runtime Metrics Layer**  
- exposes a unified API for event publication and subscription  

It is the subsystem that ensures NDH’s runtime is **reactive, modular, and extensible**.

---

## **3. Event Categories**

Events in NDH fall into six major categories:

### **3.1 State Events**
Changes in NDH’s semantic‑geometric state:

- geometry updates  
- field changes  
- domain drift  
- entity migration  

### **3.2 Stability Events**
Generated by the **Stability Auditor**:

- marginal stability  
- critical instability  
- collapse predictions  

### **3.3 Flux Events**
Generated by the **Semantic Flux Integrator**:

- flux spikes  
- overload warnings  
- damping/amplification transitions  

### **3.4 Transport Events**
Generated by the **Transport Dynamics API**:

- corridor routing  
- channel activation  
- paradox front propagation  

### **3.5 Governance Events**
Generated by governance and policy subsystems:

- policy decisions  
- governance coherence changes  
- emergency governance triggers  

### **3.6 Axiom Events**
Generated by the **Axiom Manager**:

- axiom birth  
- fusion  
- split  
- drift  
- rupture warnings  

These categories define the event ecosystem.

---

## **5. Interfaces**

### **Inputs**
- events from all runtime subsystems  
- subscription requests  
- configuration (priority rules, safety strictness)  

### **Outputs**
- routed events  
- event logs  
- replay buffers  
- safety alerts  
- ordering guarantees  

---

## **6. Subscription Model**

The Event Bus supports:

- **direct subscription** (specific event types)  
- **pattern subscription** (semantic filters)  
- **priority subscription** (critical events only)  
- **domain‑scoped subscription** (events from specific domains)  
- **flux‑scoped subscription** (events tied to semantic flux regions)  

This makes NDH’s event system **semantic‑aware**, not just structural.

---

## **7. Configuration**

Key parameters:

- **event_priority_rules**  
- **ordering_guarantees** (strict, relaxed)  
- **safety_strictness**  
- **buffer_size**  
- **logging_level**  
- **replay_mode**  

---

## **8. Safety Integration**

The Event Bus enforces:

- priority routing for critical stability events  
- throttling of high‑volume flux events  
- isolation of unsafe transport events  
- emergency governance triggers  
- rollback signals to the **State Serializer**  

If unsafe conditions arise:

- the bus halts routing  
- emits a critical alert  
- triggers scheduler safety mode  
- logs the event sequence for replay  

---

## **9. Lifecycle**

1. **Initialize** event categories and routing rules  
2. **Receive** events from subsystems  
3. **Categorize** and **prioritize**  
4. **Route** to subscribers  
5. **Log** and **buffer**  
6. **Emit** replay updates  
7. **Await** next event
